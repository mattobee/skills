# Sources

Design decisions and where they come from. The agent never loads this file — it's for humans reviewing the skill.

## Separate tester from coder

A coder testing its own work has confirmation bias. [AgentCoder](https://arxiv.org/abs/2312.13010) (2023) demonstrated this by separating code generation and test generation into distinct agents, achieving 96.3% on HumanEval vs 90.2% prior best.

## Plan-implement-verify core team

Multiple systems converge on this three-role pattern: MetaGPT, AgentCoder, ChatDev. Anthropic's [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) describes the same evaluator-optimizer loop.

## Gate each step with deterministic checks, not LLM self-evaluation

Accuracy compounds across steps — even high per-step accuracy degrades fast over long chains. The Agent Skills spec arrives at the same conclusion with its [validation loops](https://agentskills.io/skill-creation/best-practices) pattern: do the work, run a validator, fix, repeat.

## Keep agent instructions short

The Agent Skills [best practices](https://agentskills.io/skill-creation/best-practices) recommend SKILL.md under 500 lines / 5,000 tokens: "Overly comprehensive skills can hurt more than they help — the agent struggles to extract what's relevant." We apply the same to agent instruction files (under 500 words each).

## Ground specialists in official docs, not training data

LLMs confidently generate outdated API calls from stale training data. The Agent Skills [best practices](https://agentskills.io/skill-creation/best-practices) call this out directly: "A common pitfall is asking an LLM to generate a skill without providing domain-specific context — relying solely on the LLM's general training knowledge." Wire in MCP servers or documentation URLs.

## Platform-agnostic core outputs

The [Agent Skills specification](https://agentskills.io/specification) is platform-agnostic by design — skills provide knowledge and workflows, not platform config. We follow this: `.agents/` and `AGENTS.md` are the universal outputs; platform-specific config is generated by detecting the environment.

## Generic model tiers instead of provider-specific names

Frontier / mid-tier / fast. Provider mapping is in `model-guide.md`. This follows the spec's portability principle — skills should work across agents and providers.

## Multi-agent isn't always better

A single well-prompted agent handles many projects. Multiple agents add value mainly when individual accuracy is low or when work streams are clearly parallelisable. [More Agents Is All You Need](https://arxiv.org/abs/2402.05120) (2024) shows gains correlate with task difficulty.

## Practitioner heuristics (no formal source)

These are experience-derived rules of thumb. They're useful but not backed by published research:

### 70/20/10 call volume split

~70% fast, ~20% mid-tier, ~10% frontier. A budgeting guideline, not a measured optimum.

### Mid-tier handles 80–90% of coding tasks

Most implementation work doesn't need frontier reasoning.

### Frontier models tend to refactor beyond scope

Widely observed, not formally studied. Constrain scope in system prompts.

### Dual-touchpoint specialists

Calling domain specialists early (planning) and late (review) catches issues cheaper than a single review gate. Logical but not benchmarked.
